## <span style="color:#0D47A1;">1. AI-Driven Test Design: Coverage & Edge Cases</span>

### <span style="color:#0D47A1;">1.1  Coverage-First Approach</span>

<table style="width:100%; border-collapse:collapse;">
<tr><td style="background:#0D47A1; color:white; padding:10px 12px; border:1px solid #BBDEFB; font-weight:bold;">Basic Prompt</td><td style="background:#0D47A1; color:white; padding:10px 12px; border:1px solid #BBDEFB; font-weight:bold;">Coverage-Driven Prompt</td></tr>
<tr><td style="background:#E3F2FD; padding:10px 12px; border:1px solid #CCC;">"Write tests for validators.ts"</td><td style="background:#E3F2FD; padding:10px 12px; border:1px solid #CCC;">"Generate tests targeting 100% branch coverage. Include every if/else path, boundary values, all format violations and null/undefined inputs."</td></tr>
<tr><td style="padding:10px 12px; border:1px solid #CCC;">Copilot decides what to test</td><td style="padding:10px 12px; border:1px solid #CCC;">You specify the coverage strategy</td></tr>
<tr><td style="background:#E3F2FD; padding:10px 12px; border:1px solid #CCC;">Might miss edge cases</td><td style="background:#E3F2FD; padding:10px 12px; border:1px solid #CCC;">Explicitly targets edge cases</td></tr>
</table>

### <span style="color:#0D47A1;">1.2  BICEP Framework for Edge Cases</span>

*Right-BICEP mnemonic from **Pragmatic Unit Testing in Java 8 with JUnit** by Jeff Langr, Andy Hunt & Dave Thomas (The Pragmatic Programmers, 2015)*

<table style="width:100%; border-collapse:collapse;">
<tr><td style="background:#00695C; color:white; padding:10px 12px; border:1px solid #80CBC4; font-weight:bold;">BICEP</td><td style="background:#00695C; color:white; padding:10px 12px; border:1px solid #80CBC4; font-weight:bold;">Example Tests for React Component Library</td></tr>
<tr><td style="background:#E0F2F1; padding:10px 12px; border:1px solid #CCC;"><strong>Boundary</strong></td><td style="background:#E0F2F1; padding:10px 12px; border:1px solid #CCC;">Input at max length, min length 0, exactly at validation thresholds</td></tr>
<tr><td style="padding:10px 12px; border:1px solid #CCC;"><strong>Inverse</strong></td><td style="padding:10px 12px; border:1px solid #CCC;">Valid input → <code>isValid: true</code> with empty errors; invalid → <code>isValid: false</code> with non-empty errors</td></tr>
<tr><td style="background:#E0F2F1; padding:10px 12px; border:1px solid #CCC;"><strong>Cross-check</strong></td><td style="background:#E0F2F1; padding:10px 12px; border:1px solid #CCC;">validateForm totals: <code>validFields + invalidFields === totalFields</code></td></tr>
<tr><td style="padding:10px 12px; border:1px solid #CCC;"><strong>Error</strong></td><td style="padding:10px 12px; border:1px solid #CCC;"><code>undefined</code>, <code>null</code>, wrong types, empty strings, whitespace-only strings</td></tr>
<tr><td style="background:#E0F2F1; padding:10px 12px; border:1px solid #CCC;"><strong>Performance</strong></td><td style="background:#E0F2F1; padding:10px 12px; border:1px solid #CCC;">Validating 1000 form entries completes under 50 ms</td></tr>
</table>

```text
# Generate BICEP tests:
(Copilot Chat, with validators.ts open)
"For #file:src/utils/validators.ts, generate Jest tests using the
BICEP framework: Boundary, Inverse, Cross-check, Error, Performance.
Use test.each() for boundary values."

# Fill coverage gaps after running Jest with coverage:
(Copilot Chat)
"Jest coverage shows lines 34–38 in validators.ts are not covered.
Generate tests that exercise exactly these error-handling paths."
```

<table style="width:100%; border-collapse:collapse;"><tr><td style="border-left:5px solid #1A73E8; background:#E3F2FD; padding:14px 18px;"><strong style="color:#0D47A1;">⭐ KEY POINT:</strong> <span style="color:#0D47A1;">BICEP gives Copilot a structured approach to edge cases. Without it, Copilot generates happy-path + obvious failures. With BICEP, you get systematic coverage that finds real bugs — especially null/undefined handling in TypeScript.</span></td></tr></table>

## <span style="color:#0D47A1;">2. Fixtures, Mocks & Test Isolation Patterns</span>

### <span style="color:#0D47A1;">2.1  Test Fixture Architecture</span>

```text
# Create shared test fixtures:
(Copilot Chat, Agent mode)
"Create src/__fixtures__/formFixtures.ts with:
- A validFormData object with all required fields present and valid
- An invalidFormData object with multiple field errors
- A makeFormData(overrides) factory function that merges partial
  overrides onto validFormData, returning a typed FormData object
Use the FormData type from #file:src/types/index.ts."
```

<table style="width:100%; border-collapse:collapse;"><tr><td style="border-left:5px solid #1A73E8; background:#E3F2FD; padding:14px 18px;"><strong style="color:#0D47A1;">⭐ KEY POINT:</strong> <span style="color:#0D47A1;">Fixture factories are the most powerful testing pattern. <code>makeFormData({ email: 'bad' })</code> creates a form valid in every way except email — test one validation rule at a time without writing a full object each time.</span></td></tr></table>

### <span style="color:#0D47A1;">2.2  Mocking External Dependencies</span>

```text
# Mock a fetch call:
(Copilot Chat)
"Generate Jest tests for #file:src/hooks/useSubmitForm.ts mocking
the global fetch. Test: successful 200 response, network error,
non-OK status code (422 with validation errors). Use jest.spyOn
on global.fetch — do not import a fetch library."

# Mock a React context:
(Copilot Chat)
"Generate tests for #file:src/hooks/useFormValidation.ts with a
small wrapper form component. Mock only external boundaries; use the
real validation utilities rather than mocking validateRequired or validateEmail."
```

<table style="width:100%; border-collapse:collapse;"><tr><td style="border-left:5px solid #C62828; background:#FFEBEE; padding:14px 18px; border:1px solid #EF9A9A;"><strong style="color:#C62828;">⚠️ WARNING:</strong> <span style="color:#C62828;">Over-mocking proves nothing. Mock at the boundary (fetch, localStorage, external APIs), test real logic with real data. Copilot will mock aggressively — including mocking your own utility functions — if you don't explicitly constrain it to mock only external boundaries.</span></td></tr></table>

## <span style="color:#0D47A1;">3. Flaky Test Triage with Copilot</span>

<table style="width:100%; border-collapse:collapse;">
<tr><td style="background:#C62828; color:white; padding:10px 12px; border:1px solid #EF9A9A; font-weight:bold;">Pattern</td><td style="background:#C62828; color:white; padding:10px 12px; border:1px solid #EF9A9A; font-weight:bold;">Root Cause</td><td style="background:#C62828; color:white; padding:10px 12px; border:1px solid #EF9A9A; font-weight:bold;">Fix</td></tr>
<tr><td style="background:#FFEBEE; padding:10px 12px; border:1px solid #CCC;"><strong>Passes alone, fails in suite</strong></td><td style="background:#FFEBEE; padding:10px 12px; border:1px solid #CCC;">Shared DOM state between tests</td><td style="background:#FFEBEE; padding:10px 12px; border:1px solid #CCC;">Add proper <code>afterEach(() => cleanup())</code> or rely on RTL's auto-cleanup</td></tr>
<tr><td style="padding:10px 12px; border:1px solid #CCC;"><strong>Fails intermittently</strong></td><td style="padding:10px 12px; border:1px solid #CCC;">Async timing — missing <code>await</code></td><td style="padding:10px 12px; border:1px solid #CCC;">Replace raw <code>waitFor</code> timeouts with <code>screen.findBy*</code> queries</td></tr>
<tr><td style="background:#FFEBEE; padding:10px 12px; border:1px solid #CCC;"><strong>Fails on CI only</strong></td><td style="background:#FFEBEE; padding:10px 12px; border:1px solid #CCC;">Locale/timezone environment difference</td><td style="background:#FFEBEE; padding:10px 12px; border:1px solid #CCC;">Mock <code>Date</code> with <code>jest.useFakeTimers()</code></td></tr>
<tr><td style="padding:10px 12px; border:1px solid #CCC;"><strong>Fails on specific date</strong></td><td style="padding:10px 12px; border:1px solid #CCC;">Date-dependent logic</td><td style="padding:10px 12px; border:1px solid #CCC;">Pin time: <code>jest.setSystemTime(new Date('2025-06-15'))</code></td></tr>
</table>

```text
# Diagnose a flaky test:
(Copilot Chat)
"#terminalLastCommand — This test passes when run alone but fails
when the full suite runs. [paste 2-3 failure outputs].
Diagnose why it is flaky and propose a fix."

# Common fixes:
expect(result).toBeCloseTo(25.333, 3)     // float precision
jest.useFakeTimers()                       // freeze timers
jest.setSystemTime(new Date('2025-06-15'))
Math.random = jest.fn(() => 0.5)          // seeded randomness
```

## <span style="color:#0D47A1;">4. Property-Based Testing & Mutation Testing</span>

### <span style="color:#0D47A1;">4.1  Property-Based Testing with fast-check</span>

This section is optional unless `fast-check` is installed in the lab repo:

```bash
npm install --save-dev fast-check
```

<table style="width:100%; border-collapse:collapse;">
<tr><td style="background:#6A1B9A; color:white; padding:10px 12px; border:1px solid #CE93D8; font-weight:bold;">Example-Based</td><td style="background:#6A1B9A; color:white; padding:10px 12px; border:1px solid #CE93D8; font-weight:bold;">Property-Based</td></tr>
<tr><td style="background:#F3E5F5; padding:10px 12px; border:1px solid #CCC;">Tests one case: <code>email = 'bad@'</code></td><td style="background:#F3E5F5; padding:10px 12px; border:1px solid #CCC;">Tests thousands: any string without <code>@domain.tld</code> (random)</td></tr>
<tr><td style="padding:10px 12px; border:1px solid #CCC;">You choose inputs</td><td style="padding:10px 12px; border:1px solid #CCC;">fast-check generates inputs automatically</td></tr>
<tr><td style="background:#F3E5F5; padding:10px 12px; border:1px solid #CCC;">Finds anticipated bugs</td><td style="background:#F3E5F5; padding:10px 12px; border:1px solid #CCC;">Finds bugs you didn't think of (empty string, unicode, very long strings)</td></tr>
</table>

```text
# Generate fast-check property tests:
(Copilot Chat)
"Generate fast-check property-based tests for validateEmail() in
#file:src/utils/validators.ts:
1. Any RFC-compliant email string → isValid: true
2. Any string without '@' → isValid: false, error message present
3. validateForm result: validFields + invalidFields always equals totalFields"
```

### <span style="color:#0D47A1;">4.2  Mutation Testing</span>

Treat this as manual mutation analysis unless your team has installed a mutation-testing runner such as Stryker. The course workflow below uses Copilot to identify likely weak assertions; it does not require extra tooling.

```text
# Manual mutation analysis with Copilot:
(Copilot Chat)
"Look at #file:src/utils/validators.ts and identify 5 mutations that
would change behavior but might NOT be caught by the current tests.
For each: what you'd change, why the tests might miss it, what test to add."
```

<table style="width:100%; border-collapse:collapse;"><tr><td style="border-left:5px solid #1A73E8; background:#E3F2FD; padding:14px 18px;"><strong style="color:#0D47A1;">⭐ KEY POINT:</strong> <span style="color:#0D47A1;">Property-based tests are Copilot's superpower for TypeScript validators. Describe the property in natural language, Copilot translates to fast-check arbitraries. Generated inputs find edge cases (empty string, Unicode characters, very long strings, numbers-as-strings) you'd never write manually.</span></td></tr></table>

## <span style="color:#0D47A1;">5. Testing Strategy Decision Framework</span>

### <span style="color:#0D47A1;">5.1  The Testing Pyramid for React Component Libraries</span>

<table style="width:100%; border-collapse:collapse;">
<tr><td style="background:#2E7D32; color:white; padding:10px 12px; border:1px solid #A5D6A7; font-weight:bold;">Level</td><td style="background:#2E7D32; color:white; padding:10px 12px; border:1px solid #A5D6A7; font-weight:bold;">What to Test</td></tr>
<tr><td style="background:#E8F5E9; padding:10px 12px; border:1px solid #CCC;"><strong>Unit (70%)</strong></td><td style="background:#E8F5E9; padding:10px 12px; border:1px solid #CCC;">Individual validator functions, custom hooks via <code>renderHook</code>, utility functions. Pure logic, real data. Fast, many.</td></tr>
<tr><td style="padding:10px 12px; border:1px solid #CCC;"><strong>Integration (20%)</strong></td><td style="padding:10px 12px; border:1px solid #CCC;">Components with real child components (no mocking). Full form render → user interaction → validation feedback loop. RTL <code>render</code> + <code>userEvent</code>.</td></tr>
<tr><td style="background:#E8F5E9; padding:10px 12px; border:1px solid #CCC;"><strong>E2E (10%)</strong></td><td style="background:#E8F5E9; padding:10px 12px; border:1px solid #CCC;">Full form submit → mock API → success/error state. 1–2 tests per critical user journey. Playwright or Cypress.</td></tr>
</table>

<table style="width:100%; border-collapse:collapse;"><tr><td style="border-left:5px solid #1A73E8; background:#E3F2FD; padding:14px 18px;"><strong style="color:#0D47A1;">⭐ KEY POINT:</strong> <span style="color:#0D47A1;">Validators and hooks are the core logic — they need 100% branch coverage + property tests + mutation analysis. Visual rendering differences are lower risk — 80% coverage with example tests is sufficient. Allocate testing effort by business risk, not by component type.</span></td></tr></table>

## <span style="color:#1A73E8;">MODULE 8 RECAP</span>

- ✅ **Coverage-first:** run `npm test -- --coverage` to find gaps, prompt Copilot for targeted tests
- ✅ **BICEP framework:** Boundary, Inverse, Cross-check, Error, Performance — systematic edge cases
- ✅ **Fixture architecture:** shared fixtures in `__fixtures__/` + factory functions (`makeFormData(overrides)`)
- ✅ **Mock strategy:** mock at boundaries (fetch, APIs), test logic with real data. Avoid mocking your own utilities
- ✅ **Flaky diagnosis:** shared DOM state, missing `await`, floats, dates → cleanup, `findBy*`, `toBeCloseTo`, `useFakeTimers`
- ✅ **Property-based:** describe rules not examples. fast-check finds Unicode, empty strings, edge cases
- ✅ **Mutation testing:** "If I add a bug, do tests catch it?" Manual analysis with Copilot
- ✅ **Testing pyramid:** 70% unit, 20% integration, 10% E2E. Effort by business risk
